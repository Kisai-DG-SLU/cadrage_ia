## Slide 6 : Approche Technique POC

**Contenu de la slide :**
Extraction d'Embeddings (Vecteurs) vs LLM Vision. Dataset : Utilisation de DeepFashion (800k images).

**Instructions de narration :**
- Présenter les deux approches techniques comparées
- Expliquer le concept d'embedding (transformation d'image en vecteur numérique)
- Comparer latence et coût entre les deux approches
- Justifier le choix de l'approche embeddings pour la performance
- Mentionner LLM Vision comme "moteur de secours" pour cas complexes

**Timing estimé :** 4 minutes
**Conseils de présentation :**
- Utiliser des analogies simples pour expliquer les embeddings
- Mettre en contraste les performances (100ms vs 2000ms)
- Souligner l'importance de la latence pour l'expérience mobile
- Préparer un schéma mental ou une démonstration visuelle

**Notes techniques :**
- Approche A (Embeddings) : ResNet / ViT, latence < 100ms, coût faible (CapEx)
- Approche B (LLM Vision) : GPT 4.1 / Claude 4.5, latence > 2000ms, coût élevé (OpEx)
- Décision : Priorité aux Embeddings pour la performance
- Backup : LLM Vision pour les cas complexes
- Principe : Deux vecteurs proches = vêtements similaires

## Slide 7 : Critères de Succès du POC (Révisés)

**Contenu de la slide :**
Star métrique "Hit 5" : Hit@5(IA) > Hit@5(baseline) + 15 points avec p < 0.05 (amélioration significative)
Baseline observée : Performance mesurée des méthodes non-IA (TF-IDF + similarité couleur) sur dataset Fashion-Insta
Latence : < 500ms (p95) pour garantir la fluidité mobile
Validation métier : > 80% des recommandations jugées pertinentes par experts mode
Reproductibilité : Code versionné, pipeline déployable, documentation complète

**Instructions de narration :**
- Présenter les critères révisés basés sur une approche empirique
- Expliquer que la baseline sera mesurée sur le dataset Fashion-Insta (Phase 1)
- Souligner l'importance de l'amélioration statistiquement significative (p < 0.05)
- Mentionner que l'objectif absolu reste > 70% de Hit@5
- Insister sur la méthodologie rigoureuse : baseline observée plutôt que théorique

**Timing estimé :** 3 minutes
**Conseils de présentation :**
- Utiliser des comparaisons claires (baseline mesurée vs amélioration requise)
- Expliquer pourquoi la latence < 500ms est critique pour le mobile
- Mentionner que la validation humaine est aussi importante que les métriques techniques
- Préparer des exemples de ce qui constitue une recommandation "pertinente"

**Notes techniques :**
- Hit@5 (Top-5 Accuracy) : Mesure standard en recommandation (recherche d'ensemble pertinent)
- Baseline : Performance mesurée empiriquement des méthodes non-IA (TF-IDF + similarité couleur)
- Critère de succès : Hit@5(IA) > Hit@5(baseline) + 15 points avec p < 0.05
- Latence cible : < 500ms (p95) pour éviter l'abandon utilisateur
- Validation métier : Panel d'experts mode pour évaluer la pertinence stylistique
- Reproductibilité : Code versionné sur Git, pipeline déployable sur Azure, documentation complète
